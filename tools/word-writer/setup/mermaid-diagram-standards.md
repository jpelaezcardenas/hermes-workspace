# Mermaid Diagram Standards

1. Only include diagrams that clarify structure, flow, sequence, or ownership.
2. Keep diagrams simple enough to fit on a Word page.
3. Avoid giant graph sprawl.
4. Use short labels.
5. Prefer top-down or left-right flows.
6. One diagram should communicate one idea.
7. If the diagram needs a paragraph to explain how to read it, simplify it.
8. Add a short caption or lead-in sentence when useful.
9. Prefer 4 to 9 nodes for most business documents.
10. If a table communicates the same thing more clearly, use a table instead.

## Best-fit use cases
- workflow steps
- approval flow
- system interaction summary
- timeline with major phases
- role handoff sequence

## Bad-fit use cases
- dense architecture maps with many cross-links
- decorative diagrams
- screenshots converted into Mermaid for no reason
