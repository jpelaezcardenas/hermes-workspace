# PPT Maker — Prompt Guide

## What this agent does

PPT Maker generates professional SettleMint PowerPoint presentations using the frozen 2026 master template and a 22-layout slide bank. It plans the deck narrative, maps content to the correct slide layouts, fills placeholders, renders Mermaid diagrams for the appendix, and builds a `.pptx` file using the canonical `build_from_config.py` pipeline.

Every deck:
- Opens with the SettleMint Cover slide
- Uses section separators between major sections
- Ends with a Thank You slide
- Appends 4–10 Mermaid diagrams after the Thank You slide (appendix only — diagrams never appear on main slides)
- Is generated fresh — no slides, diagrams, or content are reused from prior decks

## When to use it

- Customer-facing sales decks, demo presentations, and proposal support slides
- Executive briefings and QBR decks
- Technical deep-dives and platform architecture overviews
- Investor or analyst presentations
- Internal all-hands, team updates, or enablement decks
- Conference and event slide sets

Do **not** use PPT Maker for proposal document responses — use `bid-manager`. Do **not** use it for Word documents — use `word-writer`.

## Required inputs

| Input | Details |
|-------|---------|
| **Deck title** | Exact title to appear on the cover slide (max ~50 characters; keep it short) |
| **Audience** | One of: `executive`, `technical`, `sales`, `investor` — drives tone, depth, and layout choices |
| **Core message** | 1–2 sentences: what the audience should leave believing or understanding. This shapes the narrative arc. |
| **Sections** | A brief list of the major sections you want in the deck (3–6 sections is typical). Each section is a content intent description — do not specify layout IDs. |

## Optional / configurable parameters

| Parameter | Default | Notes |
|-----------|---------|-------|
| **Slide count** | 12–15 | Specify a target if you have a strict limit. The builder may exceed it by up to 2 slides for structural reasons. Appendix Mermaid diagrams are not counted in the main total. |
| **Appendix diagrams** | Included (4–10 diagrams auto-selected) | To **opt out** of appendix diagrams, explicitly state `no appendix diagrams`. Default is to include them. |
| **Tone** | Matches audience type | Override with: `formal`, `conversational`, `data-heavy`, `story-led` |
| **Content blocks** | Agent selects from `content/` folder and DALP knowledge | Specify focus areas if relevant: e.g. `focus: compliance and identity`, `include: reference projects from GCC` |
| **Special slide types** | Agent decides based on content | Request explicitly if needed: `include a comparison table`, `include a quote slide`, `include a timeline slide`, `include a 2×2 grid for [topic]` |
| **Cover subtitle** | Auto-generated | Provide a subtitle if you want exact wording on the cover |
| **Speaker notes** | Not generated by default | Request `include speaker notes` if you want talking-point bullets added to each slide |

## Full example prompt

```
Please build a deck with the following brief:

Title: DALP — Digital Asset Lifecycle Platform
Audience: executive
Core message: DALP gives financial institutions everything they need to issue, manage, 
and service tokenised assets compliantly — without building from scratch or hiring a 
blockchain team.

Sections:
1. The tokenisation opportunity — why now, what institutions are doing
2. Why building in-house fails — time, cost, compliance complexity
3. What DALP does — the five lifecycle pillars in plain terms
4. Proof points — 3 reference clients from banking (include OCBC, Maybank, and one EU bank)
5. How we work together — implementation approach and timeline
6. Next steps

Slide count: 14 (plus appendix)
Tone: formal but not dry; this is a C-suite first meeting
Content: prioritise compliance and custody messaging
Special slides: include a timeline slide for the implementation section
Appendix diagrams: yes (include architecture and lifecycle flow diagrams)
```

## ⚠️ Important

- **Send everything in ONE message.** Provide the title, audience, core message, sections, and all optional parameters together. The agent plans and builds in one pass.
- **Do not send follow-up messages mid-task.** The agent will confirm when the PPTX is ready.
- **Describe content intent, not layout IDs.** Say "a summary of three themes" not "use slide-005". The agent selects layouts from the slide bank based on content intent. Hardcoding layout numbers bypasses the agent's planning logic.
- **The agenda slide is table-of-contents only.** It lists section names, nothing else. Do not ask for substantive content (features, bullet points, metrics) on the agenda slide.
- **Mermaid diagrams go in the appendix only.** They never appear on main slides. This is not configurable.
- **The Thank You slide has no text.** It is intentionally blank. Do not ask for content on it.
- **The PPTX is the deliverable.** A markdown outline or config JSON is not the final output. Wait for the `.pptx` file in `output/`.
- **Diagrams are generated fresh.** No cached or previously used diagrams are reused. Every Mermaid diagram is specific to this deck.
