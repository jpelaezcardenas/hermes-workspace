# Proposal Structure Compliance Checklist

This file defines the expected structure for proposals generated by bid-manager. Use it to validate that proposals contain all required sections in the correct order.

---

## 1. Expected Standard Sections

A complete DALP proposal should contain these sections in order:

| # | Section | Required | Description |
|---|---------|----------|-------------|
| 1 | Cover Page / Title | Required | Client name, proposal title, date, version |
| 2 | Executive Summary | Required | 1-page overview of value proposition, key differentiators, and solution summary |
| 3 | Understanding Client Needs | Required | Demonstrates understanding of client's specific challenges and requirements |
| 4 | Proposed Solution | Required | Detailed description of the proposed solution and approach |
| 5 | Technical Approach / Architecture | Required | Technical details, architecture, integration approach |
| 6 | Implementation Plan | Required | Phases, timeline, milestones, deliverables |
| 7 | Team & Capabilities | Required | Team experience, relevant projects, certifications |
| 8 | Experience & References | Required | Case studies, client references, relevant experience |
| 9 | Commercial Offer | Required | Pricing, commercial terms, value for money |
| 10 | Compliance & Security | Required | Security measures, compliance certifications, data handling |
| 11 | Risk Management | Optional | Risk identification and mitigation approach |
| 12 | Support & Maintenance | Optional | Post-deployment support, SLA, maintenance terms |
| 13 | Appendices | Optional | Additional supporting material, detailed technical specs |

---

## 2. Section Ordering Rules

### Mandatory Order
1. Cover → Executive Summary → Understanding → Solution → Technical → Implementation → Team → Experience → Commercial → Compliance
2. Supporting sections (Risk, Support) should appear after core sections but before Appendices
3. Appendices always come last

### Acceptable Variations
- "Understanding Client Needs" may be combined with "Executive Summary" in condensed formats
- "Technical Approach" and "Proposed Solution" may be merged into "Solution Overview"
- "Experience & References" may be split into separate sections

### Structural Anti-Patterns
- Executive Summary appearing after detailed sections
- Commercial/Price appearing before solution is established
- Compliance/Security buried in appendices
- Implementation details appearing before understanding is demonstrated

---

## 3. Required vs Optional Sections

### Always Required (Missing = Fail)
- Executive Summary
- Understanding Client Needs
- Proposed Solution
- Commercial Offer
- Compliance & Security

### Required for Full Submissions (Missing = Major Flag)
- Technical Approach / Architecture
- Implementation Plan
- Team & Capabilities
- Experience & References

### Optional But Expected in Strong Proposals
- Risk Management
- Support & Maintenance
- Case Studies (beyond references)

### Optional / Context-Dependent
- Appendices
- Detailed Pricing Breakdowns
- Technical Deep-Dives

---

## 4. Flag Conditions

### Critical Flags (Automatic Review Fail)
- [ ] Executive Summary missing entirely
- [ ] No section addressing client requirements/needs
- [ ] No commercial/pricing section
- [ ] No compliance or security section

### Major Flags (Significant Issues)
- [ ] Sections out of required order (e.g., price before solution)
- [ ] Missing both "Team" and "Experience" sections
- [ ] No implementation timeline or approach
- [ ] Executive Summary appears after section 3

### Minor Flags (Warnings)
- [ ] Section not clearly labeled (hard to locate)
- [ ] Content overflows into wrong sections
- [ ] Inconsistent heading hierarchy
- [ ] Missing optional but expected sections (Risk, Support)

---

## 5. Unexpected Content Detection

### Red Flags for Unexpected Content
- Internal DALP process descriptions not appropriate for client-facing proposals
- Team size, headcount, or organizational structure details
- Internal tooling, AI workflows, or proprietary methodologies exposed
- Revenue figures, investor information, or financial details
- Code snippets, file paths, or technical implementation leaks
- Competitive attacks or disparaging remarks about competitors
- Overblown superlatives without evidence ("world-leading," "revolutionary")

### Expected Content by Section

**Executive Summary:**
- Value proposition in 1 sentence
- Key differentiators (3-5 points)
- Summary of proposed approach
- High-level commercial indication

**Understanding Client Needs:**
- Client's stated challenges
- Root cause analysis
- Implications if not addressed
- Success criteria from client perspective

**Proposed Solution:**
- How solution addresses each client need
- Key capabilities leveraged
- Benefits quantified where possible
- Differentiation from alternatives

**Technical Approach:**
- Architecture overview
- Integration points
- Data handling approach
- Security measures

**Implementation Plan:**
- Phased approach
- Timeline with milestones
- Resource requirements
- Dependencies and assumptions

**Team & Capabilities:**
- Key team members (named)
- Relevant experience
- Certifications and credentials

**Experience & References:**
- Relevant case studies (anonymized or with permission)
- Client references
- Industry experience

**Commercial Offer:**
- Pricing structure
- Payment terms
- Value for money statement

**Compliance & Security:**
- Security certifications
- Data protection measures
- Regulatory compliance
- Privacy approach

## 7. Quick Validation Check

Before finalizing review, verify:

- [ ] All required sections present
- [ ] Sections in correct order
- [ ] Each section has meaningful content (not placeholder)
- [ ] No unexpected internal/competitive content
- [ ] Heading hierarchy consistent throughout
- [ ] Cross-references accurate if used
