# OfficeDev findings for PPTX / OOXML work

Generated: 2026-03-15

## Executive summary

I reviewed the `OfficeDev` GitHub organization and pulled a full repo inventory from the GitHub API (`450` repos at time of review). For a Python-based PPTX generator that writes raw PresentationML, the repos that actually matter are a much smaller set:

1. **`dotnet/Open-XML-SDK`** *(historically OfficeDev; now lives under `dotnet`)* — the single most valuable source. It contains:
   - the official part definitions for `PresentationPart`, `SlidePart`, `SlideMasterPart`, etc.
   - the generated class model that mirrors valid OOXML structure
   - the schema metadata used by the SDK generator
   - the built-in validator implementation
   - lots of PPTX test assets
2. **`OfficeDev/open-xml-docs`** — the best human-readable reference for PowerPoint/Open XML package structure and the minimal valid part graph for presentations.
3. **`OfficeDev/Open-Xml-PowerTools`** — archived, but extremely useful for real-world patterns: slide merging, rel-id remapping, copying dependent parts, and handling media/charts/themes safely.
4. **`OfficeDev/Open-XML-Package-Editor-Power-Tool-for-Visual-Studio`** — useful as a reference for package/part/relationship inspection logic.
5. **`OfficeDev/Office-OOXML-EmbedAddin`** — useful because it contains code generated by the old Open XML Productivity Tool; good for seeing how Microsoft-generated Open XML object trees are assembled.
6. **PowerPoint add-in samples** — useful mostly for boundary conditions and interoperability, not for raw PresentationML generation.

Bottom line: **build the helper layer primarily from `Open-XML-SDK` part/schema metadata + `open-xml-docs` package structure examples + `Open-Xml-PowerTools` relationship-copying patterns.**

## Relevant repos and what to study

## 1) `dotnet/Open-XML-SDK` (official SDK; formerly under OfficeDev)

**Repo:** https://github.com/dotnet/Open-XML-SDK

### What it does

Microsoft’s official .NET SDK for Open XML documents. This is the most authoritative implementation artifact for how Microsoft models valid Office packages, parts, relationships, and element trees.

### Why it matters for a raw-XML PPTX generator

Even if we are not using .NET, this repo gives us:

- the canonical **part graph** for PPTX packages
- the **relationship types**, **content types**, and **target paths** for each part
- the structure of **valid root elements** and child part attachments
- the **validator** implementation and associated schema metadata
- lots of test PPTX assets to diff against our generated files

### Key files/directories worth studying

#### Packaging and part definitions
- `src/DocumentFormat.OpenXml/Packaging/PresentationDocument.cs`
  - API entry point for creating/opening PPTX packages.
- `data/parts/PresentationDocument.json`
- `data/parts/PresentationPart.json`
- `data/parts/SlidePart.json`
- `data/parts/SlideMasterPart.json`
- `data/parts/SlideLayoutPart.json`
- `data/parts/ThemePart.json`
- `generated/DocumentFormat.OpenXml/.../Part_PresentationDocument.g.cs`
- `generated/DocumentFormat.OpenXml/.../Part_PresentationPart.g.cs`
- `generated/DocumentFormat.OpenXml/.../Part_SlidePart.g.cs`

These `data/parts/*.json` files are gold. They tell you, in machine-readable form:

- root type (`CT_Presentation`, `CT_Slide`, etc.)
- content type
- relationship type
- target name/path
- allowed child parts
- multiplicity and fixed-content expectations

Example from `data/parts/PresentationPart.json`:
- root: `p:CT_Presentation`
- relationship type: `.../relationships/officeDocument`
- target: `presentation`
- path: `ppt`
- child parts include `SlidePart`, `SlideMasterPart`, `ThemePart`, `NotesMasterPart`, etc.

Example from `data/parts/SlidePart.json`:
- content type: `application/vnd.openxmlformats-officedocument.presentationml.slide+xml`
- relationship type: `.../relationships/slide`
- target: `slide`
- path: `slides`
- child parts include `ImagePart`, `ChartPart`, `NotesSlidePart`, `SlideLayoutPart`, `ThemeOverridePart`, `EmbeddedPackagePart`, etc.

#### Validator and schema model
- `src/DocumentFormat.OpenXml.Framework/Validation/OpenXmlValidator.cs`
- `data/schemas/`
- `data/schemas/schemas-microsoft-com_office_powerpoint.json`
- many `generated/DocumentFormat.OpenXml.Linq/.../Linq.*.g.cs`

The SDK no longer stores raw `.xsd` as the main modeling source here; instead it uses **JSON schema metadata** and generated code. That is actually better for us, because it’s already normalized into a model we can consume.

#### Test assets and fixtures
- `test/DocumentFormat.OpenXml.Tests.Assets/assets/TestDataStorage/.../PowerPoint/*.pptx`
- `test/DocumentFormat.OpenXml.Tests.Assets/assets/TestDataStorage/.../Graphics/*.pptx`
- `samples/Linq/SvgExample/Content/EmptySlide.pptx`

These are useful as “known-good” PPTX packages for diffing:

- unzip them
- compare `[Content_Types].xml`, `.rels`, `ppt/presentation.xml`, `ppt/slides/slideN.xml`
- look at edge cases: charts, media, notes, animation, color maps, backgrounds, paragraph properties, etc.

### Useful patterns and construction techniques

#### Pattern: model package structure from part metadata, not ad hoc string-building
The SDK’s `data/parts/*.json` effectively acts like a manifest of the package graph. We should steal that idea.

**Recommendation:** build a Python registry like:

```python
PARTS = {
  "presentation": {
    "content_type": "application/vnd.openxmlformats-officedocument.presentationml.presentation.main+xml",
    "rel_type": "http://schemas.openxmlformats.org/officeDocument/2006/relationships/officeDocument",
    "path": "ppt/presentation.xml",
    "children": ["slide_master", "slide", "theme", ...],
  },
  ...
}
```

#### Pattern: separate package topology from XML payload generation
The SDK separates:
- package open/create logic
- part definitions
- root element classes
- validation

Do the same in Python:
- `package.py` — OPC zip + rel management
- `parts.py` — part metadata/constants
- `presentationml.py` — XML factories/builders
- `validate.py` — structural checks + optional schema checks

#### Pattern: validator after every major write
`OpenXmlValidator` exists for a reason. Our Python builder should have:
- a lightweight internal validator for required parts/rels/elements
- optional external validation via .NET Open XML SDK in CI or test fixtures

### Useful patterns and XML construction techniques

#### Pattern: minimal valid presentation needs more than `presentation.xml`
The docs explicitly say a usable valid presentation needs at least:
- presentation part
- slide part(s)
- slide layout part
- slide master part
- theme part
- usually notes-related defaults as well depending on scenario

That’s critical. A naive generator that only writes `ppt/presentation.xml` and `ppt/slides/slide1.xml` will often produce something Office can “repair.”

#### Pattern: initialize the root structure in the correct order
The sample `create_by_providing_a_file_name/cs/Program.cs` constructs:
- `SlideMasterIdList`
- `SlideIdList`
- `SlideSize`
- `NotesSize`
- `DefaultTextStyle`

Then wires parts in a predictable way:
- create slide
- create slide layout
- create slide master
- create theme
- attach slide layout to slide master
- attach slide master to presentation
- attach theme

That ordering is worth mirroring in our builder API.

#### Pattern: title/body shapes are not magical
The insert-slide sample shows the explicit element tree for a slide:
- `Slide`
- `CommonSlideData`
- `ShapeTree`
- non-visual group props
- group shape props
- `Shape` with `NonVisualShapeProperties`
- `PlaceholderShape(Type=Title)` for title
- `TextBody -> BodyProperties -> ListStyle -> Paragraph -> Run -> Text`

That is exactly the kind of XML factory we should create in Python.

## 3) `OfficeDev/Open-Xml-PowerTools`

**Repo:** https://github.com/OfficeDev/Open-Xml-PowerTools

### What it does

Archived library of practical Open XML utilities built on top of the SDK. It covers merging documents/presentations, updating charts, regex replacement, and other real-world tasks that require handling dependencies correctly.

### Why it matters for PPTX generation

This is the best source I found for **how to safely copy or compose slides across packages** without breaking rels, media refs, charts, themes, and embedded objects.

### Key files/directories worth studying

- `OpenXmlPowerTools/PresentationBuilder.cs`
- `OpenXmlPowerTools.Tests/PresentationBuilderTests.cs`
- `NewPptxPresentations/*.pptx`
- `NewPptxPresentations/*.txt`
- `TestFiles/` *(contains test presentations and fixtures; worth inspecting locally if cloned)*

### Useful patterns and XML construction techniques

#### Pattern: central map of elements that carry relationship IDs
`PresentationBuilder.cs` contains a `RelationshipMarkup` dictionary mapping XML element names to the relationship-bearing attributes that must be remapped when slides/parts are copied.

Examples from that mapping include:
- `a:blip -> r:embed, r:link`
- `c:chart -> r:id`
- `p:oleObj -> r:id`
- `p:contentPart -> r:id`
- media and VML relationship attributes

This is extremely useful for our helper layer.

**Recommendation:** create a Python table of all OOXML elements/attributes that can carry rel IDs, then run a generic remap pass when cloning/importing XML.

#### Pattern: slide composition should be dependency-driven, not file-copy-driven
PowerTools treats slide assembly as:
1. copy XML
2. inspect XML for relationship references
3. copy dependent parts
4. remap relationship IDs
5. preserve or merge masters/layouts/themes depending on source

That is the right mental model for any “import slide” or “duplicate slide” feature.

#### Pattern: use known-good starter presentations
`NewPptxPresentations/` contains ready-made starter decks:
- `Empty.pptx`
- `FiveSlides.pptx`
- `TenSlides.pptx`
- `TitleLayout.pptx`
- `TitleAndContentLayout.pptx`
- `BlankLayout.pptx`
- `TwoContentLayout.pptx`
- several theme/layout variants

For our Python builder, these are useful for two strategies:
- **template-first** generation: start from a minimal valid deck and mutate
- **fixture diffing**: compare generated output to a known-good reference

## Key XML / packaging patterns to incorporate into our OOXML helper layer

## 1) Part registry driven by metadata
Steal this directly from `Open-XML-SDK`.

Build a Python registry for each part containing:
- content type
- relationship type
- package path template
- root element QName
- allowed child parts
- whether part content is fixed/generated

This will prevent a lot of invalid package construction.

## 2) Explicit package topology builder
Create a `PresentationPackageBuilder` that handles:
- `[Content_Types].xml`
- `/_rels/.rels`
- `ppt/presentation.xml`
- `ppt/_rels/presentation.xml.rels`
- slide parts and slide rels
- master/layout/theme wiring
- optional notes/comment/media/chart parts

Do **not** let random feature code write files directly into the zip without registering the part graph.

## 3) XML factories for common slide constructs
Implement canonical factories for:
- empty slide shell
- title placeholder shape
- body placeholder shape
- text box shape
- picture shape
- chart frame / graphic frame shell
- table shell
- group shape shell

Mirror the nesting shown in `open-xml-docs` samples:
- `p:sld`
- `p:cSld`
- `p:spTree`
- non-visual props
- transform/group props
- shape/pic/graphicFrame nodes

## 4) Relationship-ID remapping engine
Steal the concept from `PresentationBuilder.cs`.

We need a generic pass that can:
- find relationship-bearing attributes (`r:id`, `r:embed`, `r:link`, etc.)
- copy dependent parts
- create new relationship IDs
- rewrite XML references consistently

This is mandatory for:
- slide duplication
- importing slides from another deck
- copying shapes with images/charts/media

## 5) Deterministic ID allocators
Create allocators for:
- `p:sldId/@id`
- shape IDs (`cNvPr/@id` etc.)
- relationship IDs (`rIdN`)
- notes/comment author IDs as needed

Make them deterministic and collision-safe.

## 6) Template-first bootstrap for masters/layouts/themes
Do **not** try to synthesize masters/layouts/themes from scratch unless absolutely necessary.

Best approach:
- keep a tiny, known-good starter PPTX
- clone master/layout/theme from it
- generate slides against those layout contracts

Use pure raw XML generation for slide contents; use template inheritance for the ugly global scaffolding.

## 7) Validation pipeline
Add three layers:

### Internal structural validation
Check for:
- required parts present
- required rels present
- referenced targets exist
- content types registered
- slide IDs unique
- rel IDs unique per part
- root element QName matches part type

### Diff-based fixture validation
Compare against known-good packages from:
- `Open-Xml-PowerTools/NewPptxPresentations`
- `Open-XML-SDK` test assets

### External SDK validation (optional but recommended)
In CI or local debug mode:
- open generated PPTX with .NET Open XML SDK
- run `OpenXmlValidator`
- surface errors back into Python tests

## 8) Inspection/debug tooling
Build a local utility inspired by Package Editor:
- `inspect-pptx parts`
- `inspect-pptx rels`
- `inspect-pptx xml ppt/slides/slide1.xml`
- `inspect-pptx validate`
- `inspect-pptx diff template.pptx generated.pptx`

Without this, debugging OOXML is a slow-motion headache.

## Concrete Microsoft patterns worth translating to Python

## Pattern A: create then wire
Seen in `open-xml-docs` samples.

Workflow:
1. create package
2. add presentation part
3. add slide part
4. add slide layout part
5. add slide master part
6. add theme part
7. write presentation XML with correct `sldIdLst` / `sldMasterIdLst`
8. save

In Python, keep this as explicit steps rather than magical side effects.

## Pattern B: build full element trees, not XML string concatenation
Seen everywhere in SDK/docs/generated code.

Equivalent Python approach:
- use lxml/ElementTree element factories
- centralize namespace/QName constants
- avoid raw string templates except for tiny fragments

## Pattern C: use Office-produced examples as ground truth
Seen implicitly in docs, Productivity Tool workflows, and PowerTools fixtures.

Practical workflow:
- create desired construct in PowerPoint manually
- unzip PPTX
- isolate changed parts
- encode as fixture
- translate to Python builder pattern

That is the fastest route for tricky constructs like:
- smart art adjacencies
- theme overrides
- media
- charts
- section properties
- comments/notes
