That means opening sections with client value, showing proof earlier, and making DALP's differentiation explicit without repeating the same claim.

## Lessons Review

### 1. Mixed-audience writing is still not embedded early enough

The lessons file is clear that proposals are read by mixed committees. Week 3 shows that this rule is understood at rewrite stage but not at draft stage. Drafts still assume the evaluator will work to extract value from the text. They should not have to. The first sentence of each section should tell a business, procurement, or compliance reader why the capability matters before explaining how it works.

### 2. Repetition remains a trust tax

Week 3 outputs handled technical boundaries honestly, but some sections still repeated the same core differentiator without new evidence. Repetition without progression reads like padding. The next iteration should state the differentiator once, then prove it differently in each section through compliance, lifecycle control, governance, or operational speed.

### 3. Composability should lead, but not dominate

The lessons file correctly identifies composability as DALP's strongest differentiator. Week 3 confirms that. However, composability should be the opening frame, not the whole argument. After the first mention, the writing should translate that flexibility into concrete evaluator outcomes such as lower migration risk, fewer redeployments, simpler governance change, and cleaner regulatory adaptation.

### 4. Proof has to appear before the reviewer asks for it

Week 2 already flagged screenshots and visual proof as too weak. Week 3 only partially improved this. The broader lesson is that proof blocks, screenshots, tables, and diagrams belong in the draft plan from the start. They should not be treated as garnish added during cleanup.

### 5. IP discipline held, but the stress test shows the margin is thinner than it looks

The final reviewed outputs remained clean, but the Week 3 low score on the IP stress test is a warning. Under speed or complexity pressure, it becomes easier to drift too close to implementation detail. The correct response is not vague writing. It is disciplined abstraction, using public-safe capability language while preserving specificity about outcomes, controls, and standards.

## Calibration Decisions for Week 4

### Scoring approach

Do not tighten the full rubric yet. Week 3 does not justify global hardening. It shows selective weakness, not overall inflation. Keep the overall target band at 38 to 44.

Tighten expectations only in the three persistent weak categories:

| Category | Week 4 expectation |
| --- | --- |
| Client Centricity | A section cannot score 3 unless its opening sentence clearly states client value or evaluator relevance |
| Visual / Screenshots | A technical or proposal-style draft cannot score 3 unless it includes at least two concrete proof assets planned or embedded |
| Competitive Edge | A section cannot score 3 unless it links DALP architecture to a meaningful buyer advantage |

### Process changes

**Client-value first sentence.** Before drafting each section, write one line answering: why does the evaluator care about this now. Keep that line, do not discard it.

**Proof plan before prose.** For any afternoon exercise, select the evidence format before drafting: screenshot, table, workflow figure, or comparison block. Each major section needs at least one proof move.

**Boundary sentence for non-native requirements.** When a requirement sits outside native platform scope, write the boundary sentence immediately after the capability statement. This keeps honesty integrated rather than bolted on.

**Editorial anti-repetition pass.** Before submission, highlight repeated differentiator phrases and replace duplicate wording with fresh evidence or cut it.

**IP pre-close scan.** The requested proposal-checker IP checklist file was not present in the workspace. Week 4 should still enforce a mandatory IP scan using the available checklist in `setup/ip-protection.md` until the dedicated checker checklist exists in a readable location.
